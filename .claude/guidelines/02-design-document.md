# 设计文档编写指南

**参考：** [开发工作流](workflow.md) 第 2 步

**工作流上下文：**
1. 第 1 步：读取 Jira 票 (参考 [Jira API Access Skill](../../.claude/skills/jira-api-access.md))
2. **第 2 步：创建设计文档** ← 本指南 ⭐
3. 第 3-7 步：编码、提交、PR

**用途：** 在正式编码前，记录需求背景、分析方案，获得团队或产品确认。

---

## 1. 设计文档位置

```
位置: design/<JIRA工单号>.md
示例: design/MOS-2590.md
```

- **一个 Jira 票对应一份设计文档**
- 文档在 `git add` 和 commit 之前创建和审查
- 设计审查通过后，才开始编码

## 3. 编写要点

### ✅ 要做的事

```
✅ 简洁明了 - 避免冗长和重复
✅ 画出流程图/架构图 - 文字 + 图表配合
✅ 列出假设 - "我们假设 X 成立"
✅ 考虑边界情况 - edge cases
✅ 考虑数据隐私/安全 - GDPR、密钥管理等
✅ 与现有代码对齐 - 遵循项目约定
✅ 考虑性能影响 - 时间/空间复杂度
```

### ❌ 避免做的事

```
❌ 太过详细 - 设计文档不是代码注释
❌ 模糊其辞 - "可能"、"也许"要么确认要么排除
❌ 忽视测试 - 测试计划要清晰
❌ 遗漏风险 - 主动识别潜在问题
❌ 架构过度设计 - 符合当前需求即可
❌ 跳过审查 - "自己认为没问题"不够，要得到批准
```

## 4. 审查标准

**设计文档需要得到以下审查通过才能开始编码：**

| 检查项 | 是否完成 |
|--------|--------|
| [ ] 问题背景清晰 | ✅ 审查人确认 |
| [ ] 需求完整 | ✅ 审查人确认 |
| [ ] 技术方案可行 | ✅ 技术负责人批准 |
| [ ] 没有明显风险 | ✅ 风险缓解有效 |
| [ ] 时间估算合理 | ✅ 团队确认 |
| [ ] 测试计划清晰 | ✅ QA 确认 |

**不通过审查的情况：**
- 问题定义不清楚 → 回到第 1 步（与产品讨论）
- 技术方案有问题 → 修改或考虑替代方案
- 风险过大 → 缓解措施不足 → 升级讨论
- 时间估算超出迭代 → 分解任务或调整范围

## 7. 何时需要设计文档

| 场景 | 需要设计文档 |
|------|-----------|
| 修复小 bug（1 小时内） | ❌ 可选 |
| 添加小功能（1-2 天） | ⚠️ 简化版本 (3-5 页) |
| 大型功能/重构（1+ 周） | ✅ 完整设计 |
| 架构变更 | ✅ 完整设计 + 架构评审 |
| 跨多个模块的改动 | ✅ 完整设计 |
| 影响生产环境的改动 | ✅ 包含灰度/回滚计划 |

## 8. 与代码审查的关系

```
设计审查 → 编码 → 代码审查
   ✅ 审批          ✅ 通过        ✅ 合并
     ❌ 拒绝          ❌ 返工        ❌ 返工

设计阶段解决问题 >> 代码阶段返工
```

**原则：** 花 2 小时在设计审查上，能省 10 小时的代码返工。

---

## 9. 常见问题

**Q: 设计文档太长了，能不能简化？**
A: 可以。对于小改动，可以写 3-5 页的简化版本。关键是清晰和完整性。

**Q: 设计改了，是否要更新文档？**
A: 是的。文档应该是最新的设计。每次大改都应该更新 + 版本号递增。

**Q: 如果设计审查被拒绝怎么办？**
A: 这是正常的。讨论、改进方案、重新审查。比上线后才发现问题要好得多。

**Q: 是否可以跳过设计审查直接编码？**
A: 对于小改动可以，但对于复杂改动，这通常会导致返工。不划算。
