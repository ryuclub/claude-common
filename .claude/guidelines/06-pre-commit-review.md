# 提交前审查清单

执行 commit 前，必须确认以下所有项目。
存在未确认项目时，禁止执行 commit。

## 1. 分支与计划确认

- [ ] 分支名符合命名规范（参照 [分支管理](03-branch-management.md)）
- [ ] 分支基于 `<base-branch>` 最新版本创建
- [ ] 存在对应的设计文档（`design/changes/<TICKET>.md` 或项目约定位置），且方案已获用户确认
- [ ] 变更内容与设计文档中的修改方案一致，无遗漏、无超出范围的修改

## 2. 变更内容确认

- [ ] 已通过 `git diff --cached` 确认全部变更内容
- [ ] 变更符合预期（不包含多余文件、调试代码、注释掉的代码）
- [ ] 不包含机密信息（.env, credentials, Token, 密钥等）
- [ ] 每个变更点都能追溯到需求或设计文档中的具体条目
- [ ] API 文档/Swagger 注释已添加/更新（如项目使用 Swagger）

## 3. 影响范围确认

- [ ] 已明确列出本次变更影响的所有模块/文件/接口
- [ ] 上下游依赖已确认（调用方、被调用方是否需要同步修改）
- [ ] 多环境配置文件的一致性已确保
- [ ] 共享组件或公共函数的变更已评估对其他调用方的影响
- [ ] 路由/接口变更时：已确认客户端兼容性

## 4. 回归确认

- [ ] 已确认不影响现有功能（逻辑回归测试）
- [ ] 代码编译成功（`go build ./...` 或项目的构建命令）
- [ ] 配置文件变更时：已验证语法正确性（JSON/YAML 等）
- [ ] 不残留 TODO、FIXME 或硬编码的临时值
- [ ] 错误处理路径已确认（异常情况下的行为是否正确）

## 5. 安全性确认

- [ ] 不引入新的安全风险（注入、未授权访问、信息泄露等）
- [ ] 权限控制和认证逻辑未被绕过或削弱
- [ ] 日志输出不包含敏感信息（参照 [编码原则](04-coding-principles.md) 的日志规范）
- [ ] SQL 查询或数据库操作已使用参数化/预处理语句

## 6. 用户审批

- [ ] 已向用户提示变更内容的审查结果（包含影响范围分析）
- [ ] 存在疑虑或风险时已提示用户，由用户判断
- [ ] 已获得用户的提交许可

## 7. 提交信息与 PR

- [ ] commit message 和 PR 内容遵循项目的 AI 行为规则（如有）
- [ ] commit message 格式符合提交规范（参照 [开发工作流](01-workflow.md) 第 6 步）
- [ ] PR 描述中包含：
  - 变更目的和背景
  - 主要改动列表
  - 影响范围分析
  - 测试确认项

## 检查清单使用建议

**自动化检查：**
- 使用 pre-commit hook 或 CI/CD 流程自动验证某些项目（编译、lint、格式化等）

**手动检查：**
- 使用本清单中的复选框，逐项确认
- 对于高风险变更，建议邀请 2+ 评审人员在 PR 阶段再次确认

**持续改进：**
- 如发现新的常见问题，考虑添加到清单中
- 定期更新清单，反映项目的最佳实践
